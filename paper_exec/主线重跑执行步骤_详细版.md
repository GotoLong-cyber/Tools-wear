# 主线重跑执行步骤_详细版

## 1. 文档目的

本文档用于固定后续主线实验的执行顺序、比较边界与判定规则，避免在重跑过程中再次混入旧协议、旧特征或测试集反向调参的问题。

当前已经确定：

- 主线特征：`pure RMS7`
- 标签聚合：`wear_agg = mean`
- 训练协议：`no validation set`
- 外层评测：`LOCO`
- 需要重新确认的仅有两部分：
  - `epoch`
  - `KNN` 超参数（尤其是 `k` 与 `beta`）

本文件的目标是把这两部分的后续执行步骤完全写清楚，然后按文档逐步落实。

## 3. 本轮实验的唯一调参边界

本轮只允许重新验证两类因素：

1. `epoch`
2. `KNN` 超参数

除此之外，不再变动：

- 特征集合
- 数据划分方式
- backbone 结构
- 外层 LOCO 测试协议

原则：

- `epoch` 的验证目的是证明固定轮数训练是否稳定
- `KNN` 超参数验证的目的是确定主结果到底应该偏 `blend` 还是 `knn-only`
- 这两项验证都不能使用 outer test 结果反向调参

---

## 4. 总执行顺序

后续工作严格按以下顺序进行：

1. 先完成 `pure RMS7 baseline` 的 `epoch` 敏感性分析
2. 再完成 `pure RMS7 + TMA` 主线下的 `KNN` source-only 选参
3. 基于选出的 `epoch / k / beta`，重跑最终三折主结果
4. 再决定论文正文主结果写法：
   - `blend` 作为默认主形式
   - 还是 `knn-only` 作为 strongest variant

顺序不能反过来。

原因：

- 如果先重跑最终结果，后面 `epoch / k / beta` 一旦改动，就又要重跑一遍
- 先把参数确认掉，后面三折主结果只用跑一次

## 5. 第一部分：Epoch 敏感性分析

### 5.1 目标

回答以下问题：

- 在 `pure RMS7` 主线下，固定 `epoch = 200` 是否仍然合理？
- `200` 是否位于一个较平稳的性能区间内？
- 如果不是，是否需要把固定轮数改为其他值？

本部分的目的不是为了“挑最优 test 数字”，而是为了判断：

**固定轮数训练在当前主线下是否稳定。**

### 5.2 本部分固定设置

- 特征：`pure RMS7`
- 标签：`mean`
- 协议：`no validation set`
- 模型：`TimerXL`
- 模块：只跑 `baseline`
- 不加：
  - `TMA`
  - `KNN`

### 5.3 首轮采样点

先跑以下 5 个 epoch：

- `100`
- `150`
- `200`
- `250`
- `300`

### 5.4 为什么先跑这 5 个点

- `100`：检验是否欠训练
- `150`：观察从欠训练到稳定区的过渡
- `200`：当前候选正式轮数
- `250`：观察是否出现平台
- `300`：观察是否开始过拟合或反弹

### 5.5 首轮只跑哪个折

先只跑：

- `fold1 (C1 + C4 -> C6)`

原因：

- `fold1` 是 hardest fold
- 如果 `fold1` 已经能看出趋势，没必要立刻跑三折
- 这样可以最省时间地判断 `epoch` 是否敏感

### 5.6 每个点需要记录的结果

每个 `epoch` 点只记录：

- `head-only RMSE`
- `head-only MAE`

不需要先跑：

- `TMA`
- `KNN`

### 5.7 判定标准

如果出现以下情况之一，则 `200` 可以保留：

1. `150~250` 之间波动很小
2. `200` 不是绝对最优，但落在一个平稳盆地区间中
3. `300` 没有明显优于 `200`

如果出现以下情况，则需要重新考虑固定轮数：

1. `100/150/200/250/300` 曲线大起大落
2. `200` 明显偏离最稳区间
3. `250/300` 稳定明显优于 `200`

### 5.8 本部分结束后要输出什么

输出一张 `epoch` 对照表，至少包括：

- `epoch`
- `fold1 RMSE`
- `fold1 MAE`

如果趋势已经清晰，则本部分结束，不再补三折。

如果趋势不清晰，再考虑补：

- `fold2`
- `fold3`

## 6. 第二部分：KNN source-only 选参

### 6.1 目标

回答以下问题：

- 在 `pure RMS7` 主线下，最优 `k` 大概取多少？
- 在 `pure RMS7` 主线下，最优 `beta` 是否接近 `1.0`？
- 最终主结果更应该写成：
  - `blend`
  - 还是 `knn-only`

### 6.2 本部分固定设置

- 特征：`pure RMS7`
- 标签：`mean`
- 协议：`no validation set`
- epoch：先用当前候选值
  - 若 `epoch` 敏感性分析后确认 `200` 合理，则继续用 `200`
  - 若 `epoch` 需要改，则先改后再跑本部分
- backbone：使用 `pure RMS7 + TMA` 主线

说明：

这里优先在 `+TMA` backbone 上做 KNN 选参，是因为最终主文里如果还要保留 `TMA` 这条模块线，那么最合理的 retrieval 选参对象就是：

- `pure RMS7 + TMA`

### 6.3 选参原则

本部分必须遵守：

- 只能用 `source-only` 方式做选参
- 不能根据 outer test 数字来反向选 `k / beta`

也就是说：

- 选参的依据来自 source-domain 内部 protocol
- 不能看 `fold1/fold2/fold3` outer test 哪个更好就改参数

### 6.4 先跑的最小网格

#### `k`

- `3`
- `5`
- `7`
- `10`
- `15`

#### `beta`

- `0.0`
- `0.3`
- `0.5`
- `0.7`
- `1.0`

### 6.5 这些参数的意义

- `k`：邻居数量
- `beta`：检索增量对最终预测的融合强度

其中：

- `beta = 1.0` 接近完全相信检索增量
- 当 `beta` 很接近 `1.0` 时，说明主结果更偏向 `knn-only`
- 当最优 `beta` 落在中间区间时，说明 `blend` 更合理

### 6.6 `late_q` 是否现在就重跑

当前建议：

- 先固定 `late_q = 0.8`
- 暂时不把 `late_q` 一起并入第一轮网格

原因：

- `k × beta` 已经足够回答最关键的问题
- 如果一开始再把 `late_q` 也扫进去，组合数会急剧膨胀
- 后续如果发现 `k / beta` 很敏感，再补扫 `late_q`

### 6.7 本部分需要记录的指标

每组参数至少记录：

- `delta-knn-only RMSE`
- `delta-knn-only MAE`
- `delta-blend RMSE`
- `delta-blend MAE`

### 6.8 本部分重点不是“谁数字最小”，而是“趋势”

需要回答：

1. 最优 `k` 落在哪个区间
2. `beta` 趋向 `1.0` 还是中间值
3. `blend` 和 `knn-only` 哪一条在当前主线下更稳

### 6.9 本部分结束后的判定逻辑

如果出现：

- 最优 `beta ≈ 1.0`

则说明：

- 主结果应更偏向 `knn-only`

如果出现：

- 最优 `beta` 稳定落在 `0.3~0.7`

则说明：

- 主结果更适合保留 `blend`

如果两者差异很小：

- 方法叙事仍然用 `blend`
- 同时在表中保留 `knn-only` 作为 strongest variant

只有在前两部分完成之后，才进行本部分。

### 7.1 最终冻结内容

- 最终 `k`
- 最终 `beta`
- 是否继续固定 `late_q = 0.8`

### 7.2 最终需要重跑的主线

围绕 `pure RMS7` 主线，重新跑三折：

1. `baseline`
2. `+TMA`
3. `+TMA+KNN`

### 7.3 最终需要整理的结果

每条线都要给出：

- `fold1`
- `fold2`
- `fold3`
- 三折平均

并记录：

- `RMSE`
- `MAE`

### 7.4 最终主表里至少要有的行

- `pure RMS7 baseline`
- `pure RMS7 + TMA`
- `pure RMS7 + TMA + KNN (knn-only)`
- `pure RMS7 + TMA + KNN (blend)`
- `full133 baseline`
- `full133 + TMA`
- `full133 + KNN`

说明：

这里 `full133` 作为强基线保留，不再作为主线调参对象。

- `epoch`

## 10. 最终执行判断

当前最小且正确的后续路线不是继续扩展更多新特征，而是：

1. 围绕 `pure RMS7` 重新确认 `epoch`
2. 围绕 `pure RMS7` 重新确认 `k / beta`
3. 再据此重跑最终三折主结果

在这三步完成之前，不建议：

- 继续扩展更多特征组合
- 提前写定主结果到底是 `blend` 还是 `knn-only`
- 提前把论文主表完全定稿

一句总结：

**后续主线执行顺序必须是：先 `epoch`，再 `k / beta`，最后三折定稿。**
