## 2. 执行摘要

### 2.1 核心结论

1. 当前**正式 A2 主线**没有发现“使用测试工况未来标签来决定对齐参数”的硬泄露。
2. 但当前训练流程中存在一个更严重的协议问题：**每个 epoch 都会访问 test_loader 并打印 Test Loss**，这会污染盲测集边界。
3. 历史 `A2 stage-align` 脚本是**未来知情型训练域重参数化**，不应进入正式论文主线。
4. 当前 `KNN / Retrieval V2.1` 的机制是“**未来增量检索 + 残差修正**”，不是简单替代 Timer 预测头，这一点是可以成立的。
5. 当前 `Feat_4` 的来源可以追溯到训练集内 Stage-B 排名，但正式 builder 已经变成**手工固化**，这在论文里需要重新补自动化 provenance。
6. 当前最适合的论文主线是：
   - 平台：`TimerXL / time-series foundation model`
   - 辅助：`A2-Random-S2`
   - 主创新：`Retrieval V2.1`

### 2.2 风险等级

1. `高风险`：训练阶段周期性访问测试集。
2. `高风险`：历史 `A2 stage-align` 脚本不满足盲式对齐叙事。
3. `高风险`：Retrieval 超参目前主要在测试折上选择。
4. `中风险`：正式特征构建脚本缺少每折自动选择链条。
5. `中风险`：距离门控版 Retrieval 使用整批测试距离分位数，不适合写成严格在线机制。
6. `低风险`：未来非 wear 协变量默认未 mask，但当前主训练/测试路径实际上未使用这些列做预测。

## 3. 关键发现与代码证据

## 3.1 高风险发现 1：训练过程中周期性访问测试集

### 证据

文件：`exp/exp_forecast.py`

1. 训练开始前同时构造 `val` 和 `test`：
   - `exp/exp_forecast.py:542-543`
2. 每个 epoch 结束时直接计算 `test_loss`：
   - `exp/exp_forecast.py:685-686`
3. 控制台打印 `Vali Loss` 与 `Test Loss`：
   - `exp/exp_forecast.py:687-689`

### 学术判定

这不一定等价于“标签泄露到训练梯度”，但已经违反了严格盲测协议。  
因为测试集被反复暴露给开发者和实验调参过程，审稿人会将其视为 **test-set contamination**。

### 影响

1. 当前“严格零泄露”表述站不稳。
2. 所有在该训练循环内产出的测试数值都不适合直接当作最终盲测结论。

### 修改建议

1. 默认删除训练环中的 `test_loader` 评估。
2. 如果仅为日志排障，新增参数 `--report_test_each_epoch 0`，且默认关闭。
3. 正式实验中规定：
   - 训练阶段只允许 `train + val`
   - 测试阶段只在训练冻结后执行一次 `test()`

## 3.2 高风险发现 2：历史 A2 stage-align 脚本不满足“盲式、自适应对齐”

### 证据

文件：`feature_alignment_diagnosis/scripts/build_a2_stage_aligned_fold1_data.py`

1. `progress_align()` 对整条曲线做 wear progress 重参数化：
   - `feature_alignment_diagnosis/scripts/build_a2_stage_aligned_fold1_data.py:48-76`
2. 直接使用整条 run 的末端磨损 `wear_mono[-1]`：
   - `feature_alignment_diagnosis/scripts/build_a2_stage_aligned_fold1_data.py:56-60`
3. 构造统一 `[0,1]` 网格进行插值：
   - `feature_alignment_diagnosis/scripts/build_a2_stage_aligned_fold1_data.py:70-75`
4. 训练 run 被改写，测试 run 保持原样：
   - `feature_alignment_diagnosis/scripts/build_a2_stage_aligned_fold1_data.py:145-150`

### 学术判定

这条脚本虽然没有读取测试集标签来拟合训练参数，但它对训练序列的每个位置使用了**整条 run 的未来磨损轨迹**。  
因此它不属于“窗口级因果一致的盲式对齐”，只能算 **post-hoc stage-aligned exploratory builder**。

### 影响

1. 不能把它写成正式 A2 方法。
2. 如果导师或审稿人看到这段代码，会直接追问“你是不是先知道了完整磨损进度，再回头改时间轴”。

### 修改建议

1. 将该脚本明确降级为“历史探索脚本”，不进入正式论文方法。
2. 在总文档、论文、答辩中只保留 `data_loader.py` 中的 A2-Random-S2。
3. 在仓库文档中加一句：
   - `build_a2_stage_aligned_fold1_data.py` is exploratory only and not part of the formal blind protocol.

## 3.3 高风险发现 3：Retrieval 超参存在测试折调参污染

### 证据

文件：`feature_alignment_diagnosis/scripts/evaluate_fold1_knn_retrieval.py`

1. 同时搜索多个 `k`：
   - `evaluate_fold1_knn_retrieval.py:433`
2. 同时搜索多个 `beta`：
   - `evaluate_fold1_knn_retrieval.py:434`
3. 按 `mae_full_raw` 选择最佳 blend：
   - `evaluate_fold1_knn_retrieval.py:457-459`

文件：`feature_alignment_diagnosis/scripts/evaluate_fold1_knn_delta_retrieval.py`

1. 同时搜索多个 `beta`：
   - `evaluate_fold1_knn_delta_retrieval.py:211-214`
2. 按测试折全曲线 MAE 选最佳：
   - `evaluate_fold1_knn_delta_retrieval.py:228-230`
3. 训练库 late-ish 子库阈值也是由当前实验参数反复试出来的：
   - `evaluate_fold1_knn_delta_retrieval.py:114-116`
   - `evaluate_fold1_knn_delta_retrieval.py:165-178`

### 学术判定

这里的问题不是“训练时偷看测试标签”，而是**开发期在测试折上选超参**。  
这属于标准的 protocol contamination，会削弱结论可信度。

### 修改建议

1. 把 `k / beta / library quantile` 固定成训练域内先验，不在 test fold 上搜索。
2. 更稳的做法：
   - Fold 内部只用 `train_runs` 做 library
   - 用 `val` 或 train-only bootstrap 选 `k/beta`
   - `test_run` 上只执行一次固定配置
3. 建议正式固定：
   - `k=5`
   - `beta=0.5`
   - late library 用 train-only quantile 规则

## 3.5 中风险发现 5：distance-gated Retrieval 不是严格在线机制

### 证据

文件：`feature_alignment_diagnosis/scripts/evaluate_fold1_knn_delta_retrieval.py`

1. 动态 beta 用的是当前测试批距离分位数：
   - `evaluate_fold1_knn_delta_retrieval.py:88-102`
2. `lo/hi` 都来自 `distances` 整体分布：
   - `evaluate_fold1_knn_delta_retrieval.py:95-96`
3. 只有 `gate_mode=distance_linear` 才触发：
   - `evaluate_fold1_knn_delta_retrieval.py:232-240`

### 学术判定

这条分支不适合写成“online adaptive gate”，因为它依赖整批测试样本的距离分布统计。  
当前正式主线默认 `gate_mode=none`，所以主线可保，但 gated 版本只能当 exploratory variant。

### 修改建议

1. 正式论文不主打 gated 结果。
2. 若要保留，需改成单窗可计算的门控规则，例如：
   - 只用当前窗的 `min_dist`
   - 与训练库固定阈值比较

## 5. KNN / Retrieval 的物理解释重述

### 5.1 代码层面的真实作用

1. TimerXL 先从目标通道 token 生成共享表示 `shared_repr`：
   - `models/timer_xl.py:103-107`
2. Retrieval 不直接回归绝对 wear，而是先构造：
   - `delta = future_wear - current_last_wear`
   - `evaluate_fold1_knn_delta_retrieval.py:155-159`
3. 预测时用训练库相似历史窗的未来增量来修正 head 的增量偏差：
   - `evaluate_fold1_knn_delta_retrieval.py:219-220`

### 5.2 物理解释

1. 小样本条件下，训练域对高磨损后段覆盖不足。
2. 基础头在 hardest fold 上不是“完全不会预测”，而是**系统性低估后续增量**。
3. Retrieval 的价值不在于取代 Timer，而在于提供“相似历史状态之后通常还会涨多少”的经验模板。
4. 因此它最适合修复：
   - small-sample
   - cross-condition
   - late-stage underestimation

## 7. 针对导师 6 个核心质疑的逐条整改建议

1. 质疑：是否用了测试集未来信息。  
   建议：训练阶段彻底移除 `test_loss` 周期监控；文档中将 `build_a2_stage_aligned_fold1_data.py` 降级为 exploratory。

2. 质疑：Alignment 是否因果一致。  
   建议：只保留 `data_loader.py` 的 train-only stride augmentation，并在代码注释中写清 `val/test stride=1`。

3. 质疑：特征是否人工试出来。  
   建议：重建从 Stage-A/Stage-B 到 `RMS7+Feat_4` 的 manifest 链，并要求主线 builder 读取 manifest 自动构造。

4. 质疑：KNN 的物理含义是什么。  
   建议：输出 `current_last / delta_head / delta_knn / blended_delta` 四组中间量，直接证明它修的是后续增量偏差。

5. 质疑：时间尺度处理是不是独立创新。  
   建议：在实验设计上把 A2 改成“为 Retrieval 提升表示可比性与检索质量”的辅助模块。

6. 质疑：单数据集和创新性不足。  
   建议：用更严的协议边界、机制图和分段误差表增强说服力，主动承认当前是单数据集 history-aware setting。

## 9. 可直接用于论文的方法说明

本文采用的对齐策略并非基于测试工况总长度、剩余寿命或未来阶段标签进行重标定，而是一个严格限定在训练域内部的盲式时间增强机制。具体而言，我们仅对训练窗口的已观测历史段计算局部磨损演化斜率，并在训练集窗口斜率分布上估计分位阈值，据此自适应决定是否对该窗口施加 `stride>1` 的时间压缩采样。该过程只依赖当前窗口可见的历史 wear 轨迹及训练域统计量，不访问测试工况的未来标签、全局进度或后验阶段信息；验证与测试阶段始终保持 `stride=1` 的原始时间顺序输入。因此，这一机制的本质不是“针对目标工况做对齐”，而是通过训练时的多步长增强提升模型对跨工况磨损演化速率变化的鲁棒性，使其在未见工况上能够更稳定地表征 early/mid 阶段，并为后续的检索增强提供更可比的表示空间。

## 10. 最终审计结论

当前项目**不是完全不可用**，但也**还不能直接宣称严格零泄露**。  
最关键的问题不在正式 A2，而在：

1. 训练过程中反复访问测试集。
2. 历史 stage-align 脚本与正式 A2 叙事混在同一仓库语境里。
3. Retrieval 超参目前主要在测试折上选择。

只要先修复这三点，再把 `Feat_4/SE1` 的来源用 train-only manifest 重新固定，当前代码架构是足以支撑“**基于 Retrieval 增强的跨工况磨损预测机制**”这一论文主线的。
